개발 중 `docs/features/` 히스토리 문서에 내용을 추가합니다.
완벽한 문서가 아니라 **빠른 기록**이 목표입니다.

## 문서 타입

| 타입           | 파일              | 기록 대상                            |
| -------------- | ----------------- | ------------------------------------ |
| `tech`         | `tech.md`         | 구현 내용, 설계 결정, 코드 변경 이유 |
| `troubleshoot` | `troubleshoot.md` | 발생한 문제, 원인, 해결책            |
| `ux`           | `ux.md`           | UX 결정, 기획 변경, 개선 사항        |

## 문서 형식

각 문서 타입의 형식은 아래 **4단계**에 그대로 정의되어 있습니다. 외부 템플릿 파일(`_templates/`)에 의존하지 않습니다.

## 절차

### 1단계: 타입 및 기능명 결정

사용자 요청 또는 인자에서 타입과 기능명을 파악합니다.
파악이 안 되면:

```
📝 어떤 내용을 기록할까요?

문서 타입:
1. tech — 구현 내용/설계 결정
2. troubleshoot — 문제/해결책
3. ux — UX 결정/기획 변경

기능명 (예: auth, dashboard, shared):
```

기능명이 없으면 현재 브랜치에서 추출합니다.
예: `feature/23-login-page` → `login-page`

### 2단계: 기존 문서 확인

`docs/features/{기능명}/{타입}.md` 존재 여부 확인:

- 존재하면: 기존 내용 확인 (맥락 파악용)
- 없으면: 폴더(`docs/features/{기능명}/`)를 만들고 4단계 형식으로 `{타입}.md`를 신규 생성합니다. 새 파일은 첫 줄에 `# {기능명} — {타입} 기록` 제목을 둔 뒤 4단계 항목을 append합니다.

### 3단계: 내용 입력

타입별 프롬프트:

**tech:**

```
✏️ 구현 내용을 간략히 적어주세요.
(구현한 것, 결정한 것, 변경한 이유 등 — 한두 줄도 충분해요)
```

**troubleshoot:**

```
✏️ 트러블슈팅 내용을 적어주세요.
현상:
원인:
해결:
```

**ux:**

```
✏️ UX/기획 관련 내용을 적어주세요.
(변경된 것, 결정된 것, 개선된 것 등)
```

### 4단계: 문서에 추가

폴더가 없으면 자동 생성:

```bash
mkdir -p docs/features/{기능명}
```

이슈번호는 현재 브랜치에서 자동 추출합니다.

**추가 형식:**

**tech.md:**

```markdown
## [YYYY-MM-DD] {제목} (#{이슈번호})

{입력된 내용}
```

**troubleshoot.md:**

```markdown
## [YYYY-MM-DD] {현상 한줄요약} (#{이슈번호})

**현상**: {현상}
**원인**: {원인}
**해결**: {해결}
```

**ux.md:**

```markdown
## [YYYY-MM-DD] {제목} (#{이슈번호})

{입력된 내용}
```

### 5단계: 완료 안내

```
✅ 기록 완료
→ docs/features/{기능명}/{타입}.md

💡 기능이 확정되면 /tech-doc {기능명} 으로 Confluence 초안을 만들 수 있어요.
```
