# Review Report Template

페르소나 반환 형식 · Severity 매핑 · 최종 MR 노트 템플릿.

---

## 페르소나 반환 형식 (각 Agent 호출 prompt에 포함)

````
다음 마크다운 형식으로만 반환하세요. 다른 산문은 추가하지 마세요.

### Findings

**Blocker (N건)**
- `file:line` — 짧은 제목
  - 무엇이 문제: 한 문장.
  - 왜 막아야 하나: 한 문장.
  - 이렇게 고치세요: 1~3줄 (필요 시 ```ts ... ``` 코드).

**Non-blocker (N건)**
(동일 구조)

**Praise (N건)**
- 잘된 점 한 줄.

**TODO (N건)**
- 후속 작업 제안 한 줄.

### 한 줄 총평
{단일 문장}
````

빈 섹션은 "(없음)"으로 표시. 추측은 "추측:"로 명시. diff에 없는 코드는 가정 금지.

---

## Severity 매핑 (합성 단계에서 적용)

| 페르소나 finding                       | 매핑                      |
| -------------------------------------- | ------------------------- |
| 🛡️ 보안 High/Critical, npm audit High+ | **Blocker**               |
| 🛡️ 보안 Medium                         | **Non-blocker**           |
| 🛡️ 보안 Low                            | **TODO**                  |
| 🏗️ FSD 역방향/동일 레이어 직접 import  | **Blocker**               |
| 🧑‍💻 TS `any` (예외 사유 없음)           | **Blocker**               |
| 🧑‍💻 컴포넌트에서 fetch/axios 직접 호출  | **Blocker**               |
| 🧑‍💻 그 외 컨벤션 위반                   | **Non-blocker**           |
| 🧑‍💻 가독성/네이밍/리팩터 제안           | **Non-blocker**           |
| 🧪 변경 대비 테스트 누락               | **Non-blocker**           |
| 🧪 회귀 포인트 미커버                  | **Non-blocker** 또는 TODO |

각 페르소나가 자체 판단한 Blocker/Non-blocker가 위 매핑과 충돌하면 **위 표 우선**.

---

## 최종 출력 (MR 노트 본문 템플릿)

```markdown
## 📋 코드 리뷰 — {branch}

> 변경 {N파일} / +{X}-{Y}줄. {코디네이터가 쓴 핵심 인상 1~2줄.}

---

### 🚨 Blocker ({합계})

#### 1. [{이모지 페르소나}] {제목}

- **위치**: `{file:line}`
- **무엇이 문제**: {1줄}
- **왜 막아야 하나**: {1줄}
- **이렇게 고치세요**:
  {코드 또는 짧은 안내}

(이하 Blocker 반복)

---

### 💡 Non-blocker ({합계})

#### 1. [{페르소나}] {제목}

- **위치**: `{file:line}`
- **무엇이 문제**: {1줄}
- **이렇게 고치세요**: {1줄 또는 코드}

---

### 👍 Praise ({합계})

- [{페르소나}] {잘된 점}

---

### 📝 TODO / 후속 ({합계})

- [{페르소나}] {제안}

---

### 페르소나별 한 줄 총평

- 🧑‍💻 시니어 개발자: {1줄}
- 🛡️ 보안 전문가: {1줄}
- 🏗️ 아키텍트: {1줄}
- 🧪 QA: {1줄}

---

_4 personas: 🛡️ 🏗️ 🧑‍💻 🧪 — all returned (parallel execution verified)_
```

**정렬 규칙**: Blocker → Non-blocker → Praise → TODO. 각 그룹 내 페르소나 우선순위: 🛡️ → 🏗️ → 🧑‍💻 → 🧪 (영향도 큰 순).

**빈 섹션**: "(해당 없음)"으로 명시. 섹션 자체를 지우지 말 것 — 검토자가 "리뷰가 빠진 건지 정말 없는 건지" 구분 가능해야 함.

**페르소나 도착 마커 (필수)**: 노트 마지막에 `_4 personas: 🛡️ 🏗️ 🧑‍💻 🧪 — all returned_` 줄을 **반드시 출력**. 4명 중 일부가 응답하지 않았다면 해당 이모지를 빼서 (`🛡️ 🏗️ 🧑‍💻 — 1 missing`) 사용자가 silent 누락을 인지하게 함.

**⚠️ GitLab 넘버링 주의 (필수)**: MR 노트의 항목 번호에 `#`을 붙이지 말 것. GitLab은 `#숫자`를 **이슈 링크**로 자동 변환하므로(`#1` → 1번 이슈) 엉뚱한 링크가 걸립니다. 위 템플릿처럼 `#### 1.`(마크다운 제목 + 숫자) 또는 `Blocker 1`로 쓰고, `#`는 실제 이슈/MR 참조(`Closes #5`)에만 사용합니다.
