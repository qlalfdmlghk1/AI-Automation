# Security & Compliance Checklist (ISMS-P)

> **리뷰 단계 전용 규칙입니다.** 이 체크리스트는 `/review`(보안 전문가 페르소나)와 `/secure-review`에서만 강하게 확인됩니다.
> 개발 중 상시 참고 규칙이 아니므로 `CLAUDE.md`·`plan-review`·일반 개발 rule에는 추가하지 않습니다.
> 규칙은 앞으로 계속 늘어날 수 있으므로 이 파일에 누적합니다.

## 적용 원칙

- 이 규칙은 **개인정보를 노출·처리하는 화면/코드**가 변경에 포함됐을 때 적용합니다. 개인정보 노출 지점이 없는 변경에는 **"해당 없음"** 으로 명시하고 넘어갑니다.
- 세부 기준(마스킹 포맷, 원문 조회 허용 권한 체계, 강조 스타일 토큰 등)은 **각 프로젝트 정책/디자인 시스템을 1순위**로 따릅니다. 프로젝트 정책이 없으면 위반으로 단정하지 말고 **"확인 필요"** 로 표기합니다 (리뷰어가 정책을 임의 확정하지 않음).
- 위반 후보는 정책성 항목이므로 `/review-converge` **자동 반영 대상에서 제외**하고 사람에게 escalate합니다.

## ISMS-P 및 개인정보보호 준수 규칙

### 1. 개인정보처리방침 표시

서비스 푸터 및 로그인 화면 내 개인정보처리방침 링크는 다른 약관 링크와 **시각적으로 구분되도록 굵게 표시**한다.

- **리뷰 판정(FE 관측 가능)**: 푸터/로그인 화면 마크업에서 개인정보처리방침 링크가 다른 약관 링크 대비 강조 처리(`font-weight` 강조 또는 프로젝트 디자인 시스템의 강조 토큰)되어 있는가.
- **확인 필요**: 프로젝트 디자인 시스템에 강조 토큰이 있으면 그 기준을 우선. "굵게"만인지, 색·밑줄 등 다른 시각적 구분도 허용인지는 프로젝트 정책을 따른다.

### 2. 개인정보 조회 제한

이름, 이메일, 전화번호 등 개인정보 **원문**은 **업무상 필요한 권한에만** 노출한다.

- **리뷰 판정(FE 관측 가능)**: 개인정보 원문을 렌더링하는 지점에 권한 가드(역할/권한 분기)가 적용돼 있는가. 권한 없는 경로에서 원문이 그대로 표시되지 않는가.
- **확인 필요**: 어떤 역할/권한이 원문 조회 가능한지의 기준은 프로젝트 권한 모델을 따른다. 권한 모델이 불명확하면 위반 단정 대신 "확인 필요"로 표기.

### 3. 개인정보 마스킹

개인정보 원문 조회 권한이 없는 사용자에게는 **마스킹된 정보**를 제공하며, 해당 처리는 **서버 또는 API에서 적용**한다.

- **리뷰 판정(FE 관측 가능) — 1차 기준**: **권한 없는 사용자에게 가는 응답(API/네트워크)에 원문이 포함되는가.** 이것이 실제 노출 지점이다. 권한 없는 응답에 원문이 실려오면 위반.
- **정황 증거(2차)**: FE가 **원문을 받아 클라이언트에서 마스킹**하는 패턴은 "원문이 이미 응답에 실려왔다"는 신호이므로 위반 후보로 본다. 단, 마스킹 함수 존재만으로 단정하지 말 것 — 권한 가드된 정당한 FE 표시일 수 있다. FE diff만으로 응답 페이로드를 단정할 수 없으면 **"확인 필요(서버 응답 점검)"** 로 escalate한다.
- **기본 마스킹 포맷(권장 기본값 — 프로젝트 정책 없을 때만 기준)**: 아래는 정책 부재 시 참고 기본값이며, **프로젝트 정책이 있으면 그쪽을 우선**한다. 포맷 위반은 위반 후보로 표기하되, 정책이 없으면 단정하지 말고 **"확인 필요"** 를 병기한다. **별표 개수로 원문 길이를 추정할 수 있게 두지 말 것.**

  | 항목     | 기본 포맷 (정책 없을 때 기준)                             | 예시                |
  | -------- | -------------------------------------------------------- | ------------------- |
  | 이름     | 성을 제외한 이름 글자 전부 마스킹(글자 수 무관). 2자 이름은 끝 글자, 외자·영문 이름은 정책 확인 필요 | `홍**`, `홍*동`     |
  | 이메일   | 로컬파트 앞 1~2자 외 마스킹(3자 미만이면 더 강하게). 도메인 부분 마스킹은 정책 따름 | `ab***@example.com` |
  | 전화번호 | 가운데 4자리 마스킹. 끝 4자리 노출 범위는 정책 확인 필요  | `010-****-1234`     |

## 스택별 판정 기준 (초안 — 확인 필요)

> 🚧 위 항목은 **FE(마크업·DOM 관측) 기준**으로 작성됐습니다. 네이티브·서버는 관측 지점이 달라 아래 기준을 적용합니다. **정본 기준은 각 스택 담당자·보안 리뷰에서 확정**하며, 프로젝트 정책이 없으면 위반 단정 대신 "확인 필요"로 표기합니다.

### Android

- 개인정보 원문 표시 지점(UI 렌더링)에 권한 가드가 적용됐는가.
- **Logcat/로그**에 원문 출력 금지(특히 릴리스 빌드).
- 로컬 저장: `SharedPreferences` 평문 민감정보 금지 → 암호화 저장(EncryptedSharedPreferences/Keystore).
- 마스킹 1차 기준은 FE와 동일 — **권한 없는 사용자 응답(API)에 원문이 포함되는가**(마스킹은 서버/API 책임).

### iOS

- 원문 표시 지점 권한 가드. `UserDefaults` 평문 민감정보 금지 → Keychain.
- 로그·크래시 리포트에 원문 노출 금지.
- 마스킹: 서버 응답 기준(위와 동일).

### 서버 (NestJS 등)

- **1차 기준을 서버에서 직접 관측**: 권한 없는 사용자 응답에 원문이 실리는가(마스킹은 서버/API 책임 — FE 클라이언트 마스킹에 의존 금지).
- 응답 직렬화·로깅에서 PII 노출 금지(로그 마스킹 적용).
- 권한별 응답 분기 — 원문 조회는 업무상 필요한 권한에만.
